# 规则索引

这是一个兼容索引。请优先加载下面这些拆分后的文档，而不要把所有规则塞进一个大文件里。

## 文档对应关系

- `用户输入模板.md`：用户可复制的输入模板、推荐模式说明和示例。
- `执行流程.md`：整体执行步骤、推荐模式必填检查、缺省反问、用户画像提取、搜索范围、候选池、输出要求。
- `筛选评分.md`：分桶、多样性协议、推荐模式评分、硬性淘汰、智能体最低门槛、后端信号、默认排除 IoT / 硬件接入类后端项目。
- `简历写法.md`：如何把项目改写成可信的项目简介、负责功能 / 技术难点和建议简历功能点。
- `输出模板.md`：最终简历包文稿的结构。
- `search_github_candidates.py`：候选池搜索脚本；用于自动搜索 GitHub、读取 README probe、去重、分桶、初筛前端 / 库 / IoT / 框架 / SDK / 桌面壳 / coding-agent 工具链等不合适项目，并生成短名单确认预览。
- `pull_github_repos.py`：最终候选源码拉取脚本；必须在任何源码验证前先运行。

## 快速规则摘要

- 先识别推荐模式，再搜索当前仓库。
- 推荐模式没有默认值；用户没给模式时，必须先反问。
- 支持 `agent-only`、`backend-only`、`mixed`、`safe-mode`、`challenge-mode`。
- 先构建多样化候选池，再做最终选择。
- 候选池优先用 `search_github_candidates.py` 生成；源码拉取前必须先输出 3-4 个短名单项目、选择理由和淘汰理由，并等用户确认方向。
- 智能体项目必须是完整业务系统，不能只是浏览器插件、薄封装、纯框架、SDK、桌面壳或 coding-agent 工具链；README probe 只服务候选筛选和项目定位。
- 后端项目必须是传统软件后端，默认排除 IoT、嵌入式、硬件接入、设备管理、工业控制类方向。
- 后端项目不应默认只选底层技术平台；要优先覆盖有业务闭环的业务项目，只降权浅层 CRUD、教程复刻和无状态流转 / 异步链路 / 失败恢复的项目。
- 后端项目必须有具体的追问点，不能只剩技术名和部署动作。
- 永远区分已有能力、建议改造、可写入简历。
- 最终入选项目必须先运行 `pull_github_repos.py` 真实拉取本地源码，再基于本地源码、配置、测试、迁移脚本或运行入口输出代码验证摘要；README / 文档、GitHub raw/API 和经验判断都不能替代本地拉取验证，也不能支撑最终“负责功能 / 技术难点”。
- 项目亮点必须体现技术难度，优先挖数据同步与一致性、MQ 异步链路、缓存与高并发、并发控制与幂等、任务调度、线程池与异步编排、流量治理、数据库优化、检索索引、权限安全、可观测性、接口治理、规则引擎、状态机、文件处理、实时通信、交易链路、Agent 工程等机制。
- 每个项目默认输出 5-6 条“负责功能 / 技术难点”，再输出“建议简历功能点（完成对应改造后可写）”，不能把建议改造写成已完成成果。
