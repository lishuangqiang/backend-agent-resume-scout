## 模式化输出要求

- `agent-only`：只在筛选阶段排除简单插件、薄封装或纯框架，不在最终文档中单独输出说明模块。
- `backend-only`：每个项目都要列出可写进简历的后端功能点，且默认排除 IoT / 硬件接入类项目。
- `mixed`：必须说明 Agent 项目和后端项目的互补关系；后端项目默认排除 IoT / 硬件接入类项目。
- `safe-mode`：必须给出 7 天落地计划和部署风险控制。
- `challenge-mode`：必须给出深度改造路线、风险提示和降级方案。

## 通用输出要求

- 尽量只使用中文。
- 每个推荐仓库都要给出链接。
- 清楚区分已有能力、建议改造、可写入简历。
- 必须输出代码验证摘要；摘要必须包含 `pull_github_repos.py` manifest 路径、本地仓库目录和 commit。负责功能必须来自本地源码证据，没有脚本拉取记录或本地源码证据的项目不得进入最终推荐；正式简历条目不要逐条写源码路径或证据标签。
- 简历功能点必须标注为“建议简历功能点（完成对应改造后可写）”，不能把建议改造写成已完成成果。
- 输出时优先给出“负责功能 / 技术难点”，再给出“建议简历功能点（完成对应改造后可写）”；前者强调已做或可讲清的工程实现，后者强调用户未来可补的改造。
- 负责功能要写出工程难点，不能只写产品动作；每个项目至少 5 条，优先 6 条。
- 负责功能要避免模板化表达；同一个项目内不得反复使用“围绕 / 基于 / 通过”等相同开头或相同语序，尤其禁止所有条目都写成“围绕 XXX ...”。
- 后端项目默认排除 IoT、嵌入式、硬件接入、设备管理、工业控制类项目，除非用户明确要求。
- 不要大段复制说明文档原文。
- 保持推荐适合在校生或早期候选人。
