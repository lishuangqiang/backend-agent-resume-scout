# 筛选评分

用于评估后端或业务型智能体项目。

## 项目定位

- `新奇项目`：通过新颖的产品形态、业务场景、智能体流程或差异化体验吸引面试官。
- `扎实项目`：暴露缓存、消息队列、搜索、一致性、并发、存储、可观测性、分布式设计、权限、调度等后端追问点；默认指纯软件后端项目，不包含 IoT、嵌入式、硬件接入、设备管理、工业控制类平台。

推荐模式必须由用户明确选择；未指定模式时先反问，不进入评分。

## 推荐模式评分偏好

- `agent-only`：只保留完整业务型 Agent 项目；必须优先命中业务数据、状态流转、持久化、工具调用、评测、用户价值中的 2-3 项；优先多角色协作、任务规划、领域数据、记忆、评测、风控、人类审批等模块齐全的仓库。
- `backend-only`：只保留传统软件后端项目；优先完整业务系统和业务平台，其次才是底层技术组件；Java / Go / Python 后端服务、分布式系统、平台工程都可选，但必须能讲清业务闭环和可追问工程复杂度；默认排除 IoT / 硬件接入类项目。
- `mixed`：必须跨 Agent 和后端两个大类；后端项目默认排除 IoT / 硬件接入类项目；避免两个项目都属于同质化业务或同一技术卖点。
- `safe-mode`：提高“复现难度低、文档清晰、依赖少、部署成本低、7 天可改造”的权重；降低需要 GPU、复杂云服务、多模型账号、K8s 集群的项目权重。
- `challenge-mode`：提高“业务闭环复杂、差异化强、架构可扩展、多 Agent / 分布式深度、实现细节密度”的权重；可以接受学习成本高，但不能推荐文档混乱或无法验证的项目。

## Agent 业务型强门槛

- `agent-only` 候选必须优先命中以下至少 2-3 项：业务数据、状态流转、持久化、工具调用、评测、用户价值。
- README probe 只用于判断项目类型、候选池分桶和排除框架 / SDK / 工具链，不能作为最终“负责功能 / 技术难点”证据。
- framework / SDK / library / toolkit / desktop companion / coding-agent 工具链默认降权或单独分桶，不与业务型 Agent 混排；用户明确要求工具链方向时除外。
- 对 coding-agent 周边、浏览器自动化壳、插件集合、模板仓库、awesome list、prompt collection 保持降权或淘汰，避免误判成业务型 Agent。

## 用户画像偏好

- 非 985/211 或背景普通：偏向新奇、差异化项目，但必须能落地。
- 学校 / 经历较强或投传统后端岗：偏向扎实后端项目。
- 目标是 Java 后端：扎实项目优先选择 Java / Spring 相关。
- 目标是智能体方向：优先选择完整业务型智能体系统，而不是通用框架或界面插件。
- 时间预算少于 1 周：可以建议用户选择 `safe-mode`，但不能在用户未确认时自动切换。
- 时间预算 1 个月以上且目标高差异化：可以建议用户选择 `challenge-mode`，但不能在用户未确认时自动切换。

## 多样性协议

不要总推荐最热门的项目。

1. 至少覆盖 4 个分桶和 6 组查询词。
2. 候选池尽量达到 20-40 个仓库。
3. 每个分桶保留前 3-5 个，但不要只按 star 数取前几名；优先保留业务闭环清晰、源码可验证、可改造空间大的项目。
4. 默认跨分桶推荐；`mixed` 必须跨 Agent 和后端两类。
5. 分数接近时，优先选择业务场景更清晰、二次改造更容易、能写出真实负责功能的项目。
6. 对反复出现的学生简历热门项目做降权。
7. 同一轮推荐中，不要让多个项目依赖同一简历卖点，例如都只讲“接入大模型”、都只讲“商城 CRUD”或都只讲“底层中间件”。
8. `backend-only` 默认不要全部推荐底层技术项目；除非用户明确要求中间件 / 基础设施，否则最终组合里应优先包含 1-2 个完整业务项目。


## Star 权重与反热门偏置

star 数只用于判断项目是否具备基本社区验证，不作为最终推荐排序的主因。

- 1k+ star 的项目已经可以进入正式候选池，并与更高 star 项目同等接受业务闭环、源码证据和简历价值评估。
- 1k-5k star 项目应被主动搜索和保留，尤其是细分业务清晰、工程链路完整、可二次改造的项目。
- 5k-20k star 项目可视为成熟候选，但不能仅因 star 更多而压过更适合简历的 1k-5k 项目。
- 20k+ star 项目只能说明知名度高，不自动加分；如果业务闭环弱、同质化强或反复出现在推荐中，应触发热门惩罚。
- 搜索时不要只使用全局 `sort=stars`；应组合使用业务关键词、细分方向、更新时间、语言 / 技术栈和 star 区间查询，例如 `stars:1000..5000`、`stars:5000..20000`。
- 最终推荐默认最多保留 1 个 20k+ star 项目；除非高星项目在业务闭环、源码证据、改造空间和组合互补性上明显优于所有中等热度候选。
- 后端项目和 Agent 项目都必须应用该规则：后端避免反复推荐少数头部商城 / 管理系统，Agent 避免反复推荐少数头部 Agent 框架或平台。
## 分桶

智能体分桶：

- 投研 / 炒股助手
- 多智能体协作
- 客服 / 销售自动化
- 数据分析智能体
- 代码修复 / 运维智能体
- 运营巡检 / 报告生成
- 企业知识流转

后端分桶：

- 即时通信
- 电商 / 交易
- 订单 / 支付 / 营销 / 结算
- 供应链 / 库存 / ERP / CRM
- 内容社区 / 问答 / 博客 / CMS
- 协作办公 / 项目管理 / 工单 / 客服
- 网盘 / 文件协作 / 知识库
- 在线教育 / 考试 / 预约 / 活动
- 监控告警
- 工作流 / 审批
- 分布式调度
- 文件存储 / 网盘
- 搜索推荐 / 内容社区
- 权限认证 / 单点登录 / 网关鉴权

业务项目优先信号：

- 有真实业务流程：下单、支付、退款、库存、结算、审批、工单、内容发布、消息通知、搜索、推荐、权限、对账。
- 有状态流转：订单状态、工单状态、审批状态、支付状态、库存锁定、任务状态、内容审核状态。
- 有异常边界：重复提交、支付回调幂等、超时取消、库存回滚、消息重试、对账补偿、权限变更、通知失败。
- 有工程深挖空间：缓存、MQ、ES、分布式锁、状态机、延迟队列、Outbox、本地消息表、任务调度、审计日志。
- 不只是管理后台 CRUD，而是能围绕一个业务链路讲清数据模型、接口设计、异步链路和失败恢复。

底层技术项目处理：

- API 网关、配置中心、调度平台、IAM、搜索引擎、流控组件、可观测性平台可以推荐，但不应默认占满所有名额。
- 如果最终推荐包含底层技术项目，必须说明它与业务项目的互补关系，并给出业务化改造场景。
- 如果用户明确要求“中间件 / 基础设施 / 框架源码”，才提高底层技术项目权重。

默认不进入最终推荐的后端分桶：

- IoT / 物联网平台
- 嵌入式 / 边缘设备管理
- 硬件接入 / 设备遥测 / 工业控制

只有当用户明确要求硬件、IoT、物联网、工业互联网、边缘设备方向时，才可以把这些分桶加入候选池，并且必须在输出中说明它偏行业 / 硬件方向。

## 评分模型

```text
总分 = 业务闭环
     + 后端深度
     + 简历可写性
     + 活跃度
     + 说明清晰度
     + 二次改造空间
     + 用户画像匹配度
     + 推荐模式匹配度
     + 实现细节密度
     + 社区验证弱加分（1k star 后封顶）
     - 热门惩罚
     - 插件 / 包装惩罚
     - 复杂度过高惩罚
     - 模式冲突惩罚
```

## 模式冲突惩罚

- `agent-only` 下，纯后端系统除非能自然改造成完整 Agent 业务系统，否则淘汰。
- `backend-only` 下，纯 Agent 框架、聊天机器人、LLM 应用模板除非有明显后端平台深度，否则淘汰。
- `mixed` 下，如果最终项目都来自同一大类，必须重新选择。
- `backend-only` 和 `mixed` 下，IoT、嵌入式、硬件接入、设备管理、工业控制类项目默认视为模式冲突，除非用户明确要求该方向。
- `safe-mode` 下，如果项目依赖复杂、文档弱、无法快速跑通，即使业务新奇也要降权或淘汰。
- `challenge-mode` 下，如果项目只适合照着部署、没有深度改造空间，即使容易落地也要降权。

## 硬性淘汰

- 简单浏览器插件、侧边栏助手、总结器、提示词管理器、单点功能界面增强。
- 只调用一次大模型接口并展示结果的薄封装。
- 只有列表、论文、脚手架、纯工具集、纯框架。
- 没有业务数据、任务状态、持久化、失败恢复的项目。
- 文档弱、运行路径不清、长期无人维护的仓库。
- 只能靠配置或部署撑起来的项目。
- 默认淘汰 IoT、嵌入式、硬件接入、设备管理、工业控制类后端项目；用户明确要求相关方向时除外。
- 不淘汰有完整业务闭环的电商、交易、社区、网盘、工单、CRM、ERP、知识库、在线教育、协作办公类项目；只淘汰浅层 CRUD、教程复刻和无法讲清业务链路的项目。

## 智能体最低门槛

- 能说清具体业务问题。
- 能命名成某个业务产品，如“AI 投研助手”“多智能体客服中心”“自动化运营分析师”“代码修复智能体平台”。
- 至少包含 4 类深度模块：角色分工、任务规划、工具调用、领域数据访问、长期 / 短期记忆、结果持久化、后台执行、权限 / 风控、可观测性、评测、人类审批。
- 能讨论数据模型、任务状态、失败恢复、并发、权限、部署。

## 扎实后端信号

- 缓存：多级缓存、穿透、击穿、雪崩、热键、本地缓存一致性。
- 消息队列：异步处理、延迟队列、重试、死信、幂等消费。
- 一致性：二进制日志、对账、补偿、最终一致性。
- 并发：分布式锁、乐观锁、库存扣减、重复提交防护。
- 搜索：索引设计、增量同步、搜索性能、索引一致性。
- 可观测性：链路、指标、日志、告警。
- 权限：角色权限、单点登录、令牌、接口签名、网关鉴权。

## 热门项目处理

- 普通商城、外卖、点评、网盘克隆等“教程复刻 / 浅 CRUD”项目要降权；但如果项目具备订单状态机、支付回调幂等、库存预占、优惠券结算、搜索索引、消息通知、对账补偿、权限审计等业务深度，不应因为它是电商 / 网盘 / 社区就降权。
- 头部智能体框架在没有业务改造时要降权。
- 同一轮会话里重复推荐过的项目要降权。
- 星标很高但业务闭环弱的项目要降权；超过 20k star 不再额外加分，且应与 1k+ 的细分项目重新比较业务价值和改造价值。
- 如果仍然推荐，要说明它为什么仍然合适，以及做了什么改造。
