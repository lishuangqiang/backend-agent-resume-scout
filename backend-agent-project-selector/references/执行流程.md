# 执行流程

在筛选后端或业务型智能体项目并生成简历包时，按以下流程执行。

## 默认假设

- 若用户未说明背景，默认视为在校生或初期后端候选人。
- 默认目标：后端实习，或智能体 / 后端实习。
- 默认输出语言：中文。
- 推荐模式没有默认值，必须由用户明确选择。
- 每次用户初始化找项目、选择项目、推荐项目或生成简历项目时，都必须先检查推荐模式。
- 用户没有给出推荐模式时，必须先完整列出 `agent-only`、`backend-only`、`mixed`、`safe-mode`、`challenge-mode` 五种模式并反问，不能只问“选哪种”，也不能自行按 `mixed`、`agent-only` 或其他模式执行。

## 输入解析

1. 先检查用户是否明确指定推荐模式：`agent-only`、`backend-only`、`mixed`、`safe-mode`、`challenge-mode`。
2. 如果没有指定推荐模式，立即停止后续流程，使用下方固定话术完整展示所有可选模式并反问用户选择哪一种模式。
3. 用户回复模式后，再提取用户画像：技术栈、目标岗位、时间预算、背景水平、项目偏好、避开方向、输出需求。
4. 如果用户画像信息不足，可以基于默认画像继续执行，但必须在输出中标注“按默认画像处理”。
5. 如果用户需要示例，加载 `用户输入模板.md`，给出可复制模板和 1-2 个贴近场景的示例。

## 缺少推荐模式时的反问

当用户初始化找项目但没有明确给出推荐模式时，只询问这一件事，不要同时开始搜索项目；必须完整展示下面 5 个模式，禁止省略模式清单：

```text
你想按哪种推荐模式执行？

- agent-only：只找完整业务型 Agent 项目
- backend-only：只找传统软件后端项目，默认排除 IoT / 硬件接入类
- mixed：一个 Agent 项目 + 一个传统软件后端项目
- safe-mode：只推荐容易落地、部署成本低的项目
- challenge-mode：推荐更难、更有差异化的项目

你回复其中一个模式后，我再开始搜索 GitHub 并生成简历写法。
```

## 推荐模式执行规则

- `agent-only`：最终推荐只保留完整业务型 Agent 项目；不得推荐浏览器插件、简单聊天机器人、薄封装、纯框架、SDK、桌面壳或 coding-agent 工具链，除非用户明确要求工具链方向。
- `backend-only`：最终推荐只保留传统软件后端项目；优先完整业务系统和业务平台，其次才是底层技术组件；默认排除 IoT、嵌入式、硬件接入、设备管理、工业控制类项目。
- `mixed`：最终推荐至少一个 Agent 项目和一个后端项目；如果用户只要 2 个项目，就按“一个 Agent + 一个后端”输出；其中后端项目默认排除 IoT、嵌入式、硬件接入、设备管理、工业控制类项目。
- `safe-mode`：优先文档清楚、依赖少、能本地或 Docker 快速跑通、7 天内可做出简历改造的项目；降低复杂分布式和重模型依赖权重。
- `challenge-mode`：优先业务闭环复杂、差异化强、可扩展为平台化系统的项目；允许部署和理解成本更高，但必须能写出清晰改造路线。

## 执行步骤

1. 确认用户已明确选择推荐模式；如果没有，先反问并停止。
2. 提取用户画像。
3. 广泛搜索：至少 6 组查询，覆盖 4 个以上分桶，不要直接选全局高星项目；`backend-only` 必须同时覆盖业务项目分桶和技术平台分桶。
4. 自动候选池：优先运行 `python backend-agent-project-selector/references/search_github_candidates.py --mode <推荐模式> --output <候选池.json> --shortlist-output <短名单预览.md>`；该脚本会搜索 GitHub、读取 README probe、去重、分桶、初筛前端 / 库 / IoT / 框架 / SDK / 桌面壳 / 工具链等不合适项目，并生成候选池 JSON。
5. 构建候选池：如果搜索结果允许，先收集 20-40 个仓库，记录链接、星标、语言、更新时间、描述、README probe 状态、分桶、初筛分和初筛理由；如果脚本结果不足或查询受限，说明原因并手动补充搜索。
6. 筛选打分：按 `筛选评分.md` 过滤和打分，保留每个分桶前 3-5 个；`agent-only` 必须优先保留至少命中业务数据、状态流转、持久化、工具调用、评测、用户价值中 2-3 项的项目；后端推荐默认剔除 IoT / 硬件接入类项目，除非用户明确要求该方向。
7. 短名单确认：在拉取源码前，必须先输出 3-4 个拟拉取项目，包含项目链接、分桶、技术栈、选择理由、主要淘汰理由和待执行的 `pull_github_repos.py --repo` 列表，并让用户确认方向。用户未确认前不要拉取源码。
8. 按推荐模式选最终项目：推荐 2-3 个；若是 `backend-only`，除非用户明确要求基础设施 / 中间件，否则不要让底层技术项目占满全部名额，优先保留 1-2 个完整业务项目。
9. 强制拉取源码：用户确认短名单后，将最终候选仓库 URL 写入临时文件或作为多个 `--repo` 参数，必须运行 `python backend-agent-project-selector/references/pull_github_repos.py --repo <url> ... --output-dir .repo-source-cache --manifest repo-source-manifest.json`；不得用 GitHub raw/API、网页、README 或模型记忆替代该步骤。
10. 核实 manifest：只允许 `repo-source-manifest.json` 中 `status` 为 `cloned` 的仓库进入源码验证；脚本失败、仓库无法拉取、本地路径不存在或源码不可读时，该候选必须淘汰。
11. 本地源码验证：只从脚本拉取到本地的仓库目录读取源码、配置、测试、迁移脚本或运行入口；README / 文档只能辅助确认项目定位，不能作为“负责功能 / 技术难点”的唯一来源。
12. 代码证据：每个最终项目至少核对 5 个关键代码证据点，例如 Controller / Service / Job / Consumer / Handler / Plugin / Entity / Mapper / Migration / 配置文件 / 测试用例；输出时要标出证据文件或目录。
13. 功能提取：负责功能必须来自本地源码证据，不能凭项目名、README 介绍、网页摘要、历史经验或模型推断猜测项目难点。
14. 技术深挖：为每个推荐项目至少挖掘 5-6 个能体现工程难度的负责功能，优先覆盖数据同步与一致性、MQ 异步链路、缓存与高并发、并发控制与幂等、任务调度、线程池与异步编排、流量治理、数据库优化、检索索引、权限安全、可观测性、接口治理、规则引擎、状态机、文件处理、实时通信、交易链路、Agent 工程等机制。
15. 生成输出：在当前工作区生成简历包文稿，按 `输出模板.md` 组织。

## 强制源码拉取脚本

- 拉取源码前必须完成短名单确认：先给用户展示拟拉取项目、选择理由、淘汰理由和待执行的仓库 URL；用户确认后才执行拉取。
- 脚本路径：`references/pull_github_repos.py`。
- 执行时机：确定最终候选后、生成任何“代码验证摘要”或“负责功能 / 技术难点”之前。
- 推荐命令：`python backend-agent-project-selector/references/pull_github_repos.py --repo https://github.com/owner/repo --output-dir .repo-source-cache --manifest repo-source-manifest.json`。
- 多仓库命令：为每个候选重复传入 `--repo`，或使用 `--repos-file <临时文件>`。
- 成功标准：manifest 中对应项目 `status` 必须为 `cloned`，且 `local_path` 指向可读取的本地仓库。
- 淘汰规则：未执行脚本、脚本失败、manifest 缺失、状态不是 `cloned`、本地源码不可读、无法找出至少 5 个源码证据点，均不得进入最终推荐。

## 输出要求

- 每个推荐仓库都要给出链接。
- 要有候选池表。
- 要有推荐模式说明、多样性说明和可替换项目。
- 要区分已有能力、建议改造、可写入简历。
- 简历条目要具体，能被追问；源码证据统一放在“代码验证摘要”，不要在正式负责功能中逐条标注“证据 / 来源”，同时必须写清实现边界，不能把未实现内容写成已有成果。
- 必须输出 5-6 条“负责功能 / 技术难点”，且每条都要体现业务场景、关键技术机制、异常处理和可追问结果；同一项目内必须主动改变开头、主语和语序，禁止批量套用“围绕 XXX ...”等固定句式。
- 必须输出代码验证摘要，说明阅读了哪些关键目录 / 文件 / 类；没有源码证据的项目不得进入最终推荐。
- 如果采用 `safe-mode`，必须给出 7 天落地计划。
- 如果采用 `challenge-mode`，必须给出深度改造路线和风险提示。
